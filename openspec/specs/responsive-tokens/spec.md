# responsive-tokens Specification

## Purpose
Token extensions: composite typography tokens, typography scale, responsive breakpoints, and breakpoint definitions.

## Requirements

### Requirement: Breakpoints token group
The design token system SHALL support a `breakpoints` group defining viewport thresholds for `@media (min-width)` queries.

- Each token has `$type: dimension` with a pixel value
- Keys are usable as references in `$extensions.responsive`
- With `DESIGNBOOK_FRAMEWORK_CSS=tailwind`, default breakpoints: `sm` (640px), `md` (768px), `lg` (1024px), `xl` (1280px) â€” user MAY customize
- Token intake workflow SHALL include a step asking about viewport breakpoints

### Requirement: Breakpoints separate from layout-width
`breakpoints` and `layout-width` are independent groups. They MAY have different values for the same key names. `breakpoints` -> `@media` resolution; `layout-width` -> container max-widths.

### Requirement: No CSS output for breakpoints
The `breakpoints` group SHALL NOT generate a `breakpoints.src.css` file. Values only appear inside `@media` rules generated by responsive tokens.

### Requirement: Responsive extensions on dimension tokens
Any token with `$type: dimension` MAY include `$extensions.responsive` â€” an object mapping breakpoint keys to dimension values. `$value` is the mobile-first base. Without `$extensions.responsive`, the token behaves as a single static CSS variable via `@theme`.

### Requirement: Stepped @media CSS generation
Responsive tokens generate CSS using `:root` and `@media (min-width)` blocks instead of `@theme`.

- One breakpoint: `:root` with base `$value` + one `@media (min-width)` block
- Multiple breakpoints: one `:root` + one `@media` per breakpoint, ordered ascending
- Mixed groups: static tokens in `@theme`, responsive tokens in `:root` + `@media`, both in the same `.src.css` file

### Requirement: Invalid breakpoint key detection
Unknown breakpoint references SHALL be skipped with a CSS comment: `/* WARNING: unknown breakpoint "<key>" */`

### Requirement: Schema validation for $extensions.responsive
`design-tokens.schema.yml` SHALL validate `$extensions.responsive`: string values pass, non-string values (numbers, objects) fail.

### Requirement: Typography composite token format
`$type: typography` tokens have an object `$value` with optional keys: `fontFamily`, `fontSize`, `lineHeight`, `fontWeight`, `letterSpacing`. At least `fontSize` required.

- Schema accepts `$value` as object, rejects plain string
- `$type: typography` and `$type: fontFamily` MAY coexist in the same group

### Requirement: DeboTypographySection renders only fontFamily tokens
`DeboTypographySection` filters by `$type === 'fontFamily'` for font cards. Count in collapsible header reflects fontFamily tokens only (ignores typography tokens). A hardcoded `TYPE_SCALE` constant provides the preview table: H1 (30/38/700), H2 (24/32/700), H3 (20/28/700), Title (18/26/700), Body (16/24/400), Small (14/20/400), Caption (12/16/400). First fontFamily token's heading font applied to preview text.

### Requirement: Typography scale tokens in design-tokens.yml
Composite `$type: typography` tokens under `semantic.typography-scale` represent semantic roles (e.g. `display-lg`, `headline-md`, `body-md`) with `fontSize`, `fontWeight`, `lineHeight`, `fontFamily` sub-values.

- Intake Step 3 (Choose Typography) asks for font families AND typography scale with semantic roles
- Stored as `semantic.typography-scale.<role>` with `$type: typography`
- fontFamily references use DTCG syntax: `{semantic.typography.heading}`

### Requirement: Skill generates type-scale tokens
`designbook-tokens` skill generates `$type: typography` composites alongside `$type: fontFamily` tokens.

- Names: `h1`, `h2`, `h3`, `body-lg`, `body`, `small`, `caption`
- Font sizes follow modular scale from 16px base; each includes `fontFamily`, `fontSize`, `lineHeight`, `fontWeight`
- Heading tokens use heading font; body tokens use body font

### Requirement: CSS expansion of typography composite tokens
The css-mapping system supports `expand: "typography"` for composite tokens.

- Expansion produces per token: `--text-{name}` (fontSize), `--text-{name}--weight`, `--text-{name}--line-height`
- `fontFamily` sub-value NOT emitted (handled by separate typography fontFamily group)
- css-mapping includes `typography-scale` group with `prefix: text`, `path: semantic.typography-scale`, `expand: typography`
- Output: `css/tokens/typography-scale.src.css` with `@theme` block containing `--text-*` properties
