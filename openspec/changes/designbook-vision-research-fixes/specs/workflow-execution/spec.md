## MODIFIED Requirements

### Requirement: Step files encapsulate CLI commands

The `designbook-workflow` skill provides four step files that workflows load by reference:

- `steps/create.md` — `workflow create` command, captures `$WORKFLOW_NAME`
- `steps/update.md` — `workflow update --status in-progress|done`
- `steps/add-files.md` — `workflow update --files <paths>` (must be called before `validate`, not on done)
- `steps/validate.md` — `workflow validate $WORKFLOW_NAME`, interpret JSON output, run fix loop

The `_debo()` bootstrap helper is scoped to a single Bash block. Each new Bash tool call starts a fresh shell — the helper MUST be redefined at the top of every block that uses it. For single-command calls, the inline pattern `eval "$(npx storybook-addon-designbook config)" && npx storybook-addon-designbook <command>` is an equivalent alternative.

#### Scenario: Bootstrap helper redefined per Bash block
- **WHEN** an agent executes a new Bash tool call
- **THEN** the `_debo()` helper MUST be redefined at the top of that block before any `_debo` call

#### Scenario: Inline eval pattern for single commands
- **WHEN** only one CLI command is needed in a Bash block
- **THEN** the agent MAY use `eval "$(npx storybook-addon-designbook config)" && npx storybook-addon-designbook <command>` instead of defining `_debo()`

#### Scenario: Files registered before validation
- **WHEN** an agent reads `add-files.md`
- **THEN** it understands that `--files` must be passed on an `--status in-progress` update, never on done

#### Scenario: Fix loop runs until exit 0
- **WHEN** `workflow validate` returns exit code != 0
- **THEN** the step instructs the agent to fix errors and re-run until all pass

## ADDED Requirements

### Requirement: Task ID naming convention

Task IDs generated by `workflow create` and intake expansion follow the pattern `<step>-<param_value>` where `param_value` preserves the original casing and spaces from intake params (e.g., `create-vision-Designbook Docs`). Agents SHALL NOT construct task IDs manually — they SHALL capture IDs from CLI responses (`workflow done`, `workflow instructions`, or error messages).

#### Scenario: Task ID includes original param value
- **WHEN** intake provides `product_name: "Designbook Docs"` and the step is `create-vision`
- **THEN** the generated task ID is `create-vision-Designbook Docs`

#### Scenario: Agent retrieves task ID from CLI
- **WHEN** an agent needs to reference a task
- **THEN** it SHALL use the ID returned by `workflow done`, `workflow instructions`, or `workflow list` — never construct it by string concatenation

### Requirement: Params required when expected_params has required entries

`workflow done --task intake` MUST include `--params` when `expected_params` (from `workflow create` response) contains any entry with `required: true`. `--params` MAY be omitted only when no entry has `required: true`. The presence or absence of `each` in stage definitions is irrelevant to this rule.

#### Scenario: Workflow with required params
- **WHEN** `expected_params` includes any entry with `required: true`
- **THEN** `workflow done --task intake --params '<json>'` MUST provide all required params

#### Scenario: Workflow without required params
- **WHEN** no `expected_params` entry has `required: true`
- **THEN** `workflow done --task intake` MAY be called without `--params`

#### Scenario: Missing required param is rejected
- **WHEN** `workflow done --task intake` is called without `--params` but required params exist
- **THEN** the CLI SHALL reject with an error listing the missing params
