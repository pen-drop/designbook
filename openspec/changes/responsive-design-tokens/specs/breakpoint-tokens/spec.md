## ADDED Requirements

### Requirement: Breakpoints token group
The design token system SHALL support a `breakpoints` token group that defines viewport thresholds for `@media (min-width)` queries.

#### Scenario: Breakpoints defined in design-tokens.yml
- **WHEN** the tokens file contains a `breakpoints` group
- **THEN** each token SHALL have `$type: dimension` and a pixel value
- **AND** the keys SHALL be usable as references in `$extensions.responsive` on other tokens

#### Scenario: Tailwind default breakpoints
- **WHEN** the user runs `/debo-design-tokens` with `DESIGNBOOK_FRAMEWORK_CSS=tailwind`
- **THEN** the workflow SHALL suggest standard Tailwind v4 breakpoints: `sm` (640px), `md` (768px), `lg` (1024px), `xl` (1280px)
- **AND** the user MAY customize or add additional breakpoints

#### Scenario: Breakpoints required in intake
- **WHEN** the token intake workflow runs
- **THEN** it SHALL include a step asking about viewport breakpoints
- **AND** breakpoints SHALL be a required token group in `create-tokens.md`

### Requirement: Breakpoints separate from layout-width
The `breakpoints` group SHALL be independent from the `layout-width` group.

#### Scenario: Different values allowed
- **WHEN** the tokens file defines both `breakpoints` and `layout-width`
- **THEN** they MAY have different values for the same key names (e.g., `breakpoints.lg = 1024px` and `layout-width.lg = 960px`)
- **AND** `breakpoints` SHALL be used for `@media` resolution
- **AND** `layout-width` SHALL be used for container max-widths

### Requirement: No CSS output for breakpoints
The `breakpoints` group SHALL NOT generate its own CSS file.

#### Scenario: Breakpoints consumed only by responsive resolution
- **WHEN** the CSS generation pipeline runs
- **THEN** the `breakpoints` group SHALL NOT produce a `breakpoints.src.css` file
- **AND** breakpoint values SHALL only appear inside `@media` rules generated by responsive tokens
