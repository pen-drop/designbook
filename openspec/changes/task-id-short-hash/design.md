## Context

Task IDs are generated by `generateTaskId()` in `workflow-resolve.ts` and used throughout the workflow lifecycle: `write-file`, `done`, `validate`, `add-file`. The current pattern `<step>-<param_value>` is fragile — agents confuse it with human-readable titles and must reconstruct it manually because the CLI doesn't return expanded task IDs after intake.

## Goals / Non-Goals

**Goals:**
- Replace `generateTaskId()` with short hash generation (4-6 hex chars)
- Return expanded task IDs in `workflow done --task intake` RESPONSE JSON
- Update `workflow-execution.md` to document the new pattern

**Non-Goals:**
- Migrating archived workflows to new ID format
- Changing how `write-file` or `done` resolve task IDs (still simple string match)
- Changing the task title format (titles stay human-readable)

## Decisions

### D1: Hash generation strategy

Generate a 6-character hex hash from `step + JSON.stringify(params) + index`. Use Node's built-in `crypto.createHash('sha256')` and take the first 6 hex characters.

The `index` (position within the expansion loop) prevents collisions when two tasks share the same step and identical params. 6 hex chars = 16M combinations, far more than any workflow needs.

**Format:** `<step-basename>-<hash>` — e.g. `create-vision-a3f2b1`

Keep the step basename as prefix for human readability in logs and error messages. The hash suffix replaces the param value.

**Alternative considered:** Pure hash without step prefix. Rejected — error messages like `Task not found: a3f2b1` are unreadable without context.

### D2: Return `expanded_tasks` in intake RESPONSE

After `runPlanLogic()` expands iterables, the RESPONSE JSON from `workflow done --task intake` includes:

```json
{
  "stage": "create-vision",
  "transition_from": "intake",
  "next_stage": "create-vision",
  "next_step": "create-vision",
  "expanded_tasks": [
    { "id": "create-vision-a3f2b1", "step": "create-vision", "stage": "create-vision", "title": "Create Vision: Designbook" }
  ]
}
```

Agents use `expanded_tasks[].id` for all subsequent CLI calls. No manual ID construction needed.

### D3: Skill documentation update

Update `workflow-execution.md` Phase 2 "Task ID Convention" to:
- State that task IDs are short hashes, not human-readable
- Instruct agents to capture IDs from `expanded_tasks` in the intake response
- Remove guidance about the `<step>-<param_value>` pattern

## Risks / Trade-offs

- [Low] Existing active workflows (mid-execution) will still have old-style IDs → Only new workflows get hash IDs; no migration needed
- [Low] Hash collisions → 6 hex chars in workflows with <50 tasks is effectively collision-free; index param adds extra safety
- [Low] Less readable IDs in `tasks.yml` → Step prefix preserves context; titles remain human-readable
