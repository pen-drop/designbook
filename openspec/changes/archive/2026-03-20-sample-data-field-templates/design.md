## Context

Sample data (`data.yml`) is generated by the `create-sample-data` task in `designbook-sample-data`. Currently, all field values are generated as plain strings regardless of the field's actual type. For backends like Drupal, many field types require structured objects — `formatted_text` needs `{ value: "...", format: "basic_html" }`, `link` needs `{ uri: "...", title: "..." }`, `image` needs `{ uri: "public://...", alt: "..." }`. Without this structure, the sample data doesn't match what Drupal actually delivers, causing rendering issues in Storybook.

The existing skill system already has a pattern for this: `view_mode` on bundles declares a `template`, and the `map-entity` stage loads rules matching `when: template: <name>`. We apply the same pattern to field-level sample data generation.

## Goals / Non-Goals

**Goals:**
- Fields in `data-model.yml` can declare `sample_template` with `template` and `settings`
- `settings.hint` guides content generation; additional settings keys are template-specific and documented per template
- Rules in skills declare `when: template: <name>` or `when: field_type: <type>` to provide output structure
- `designbook.config.yml` gains `sample_data.field_types` — a project-level map from field type to template name
- During data model creation, the AI reads this config and auto-assigns `sample_template` to matching fields
- `designbook-sample-data-drupal` skill provides rules for common Drupal field types

**Non-Goals:**
- Runtime validation of sample data structure against field type (stays as warnings only)
- Auto-discovery of templates without explicit config or `sample_template`
- Changing how reference fields are resolved

## Decisions

### Decision: `sample_template` as explicit field key, not auto-derived at generation time

Auto-deriving at generation time (reading `field_type` from config on-the-fly) would make `data.yml` generation non-deterministic and harder to inspect. Storing `sample_template` explicitly in `data-model.yml` makes the model self-documenting and gives users a clear override point.

**Why not derive at generation time?** The model would be less readable — you'd need to cross-reference config to understand why a field gets a structured value.

### Decision: Config-based auto-assignment during model creation, not hardcoded rules

`sample_data.field_types` in `designbook.config.yml` maps field types to template names. The data-model creation rule reads this and sets `sample_template` automatically. This keeps backend-specific mappings out of the core skill and fully configurable per project.

**Alternative considered**: Hardcode Drupal mappings in `designbook-data-model-drupal/rules/`. Rejected because it's inflexible — projects using custom field types or different naming couldn't override it.

### Decision: Reuse existing `when` condition system for template rules

Rules use `when: template: <name>` or `when: field_type: <type>` as conditions. No new file discovery mechanism (no `field-sample/` directory). This is consistent with how `view_mode` templates work in scenes.

`field_type` in `when` serves as a fallback for models that predate `sample_template` — the AI applies the matching rule even without an explicit `sample_template` on the field.

### Decision: `designbook-sample-data-drupal` as new skill

Drupal-specific field templates belong in a Drupal-specific skill, following the pattern of `designbook-scenes-drupal`. The core `designbook-sample-data` skill stays backend-agnostic.

## Risks / Trade-offs

- **Risk**: Models created before this change have no `sample_template` → Mitigation: `field_type` condition in rules provides automatic fallback coverage
- **Risk**: `sample_data.field_types` config not set → Mitigation: the rule documents that without config, no auto-assignment happens; users can always add `sample_template` manually
- **Trade-off**: `sample_template` in data-model.yml adds verbosity for complex field types — acceptable because it makes the model self-documenting and removes ambiguity
