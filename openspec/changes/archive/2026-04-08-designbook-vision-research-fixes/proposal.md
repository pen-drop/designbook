## Why

The `--research` audit of the `vision` workflow revealed three friction points in `workflow-execution.md` that caused retries and ambiguity during execution, plus one duplication between a rule and its task file. These issues affect every workflow that uses the documented patterns, not just `vision`.

## What Changes

- **Fix `_debo()` bootstrap documentation**: The helper function pattern in Phase 0 does not work reliably because each Bash tool call starts a fresh shell — `eval` inside a function does not persist `DESIGNBOOK_HOME` across calls. Document the inline `eval && ...` pattern as the canonical approach.
- **Document task ID naming convention**: Task IDs generated by `workflow create` use the pattern `<step>-<product_name>` with original casing and spaces. The current docs give no guidance, causing agents to guess wrong IDs and rely on error messages.
- **Clarify singleton vs. required-params workflows**: The "Singleton Workflows" section states that `workflow done --task intake` can be called without `--params`, but workflows with `expected_params` where `required: true` (like `vision`) still need `--params` even without `each`. Clarify the distinction.
- **Remove duplication in vision-format rule**: `vision-format.md` repeats constraints (Problems & Solutions format, Key Features as bulleted list) that are already specified in `create-vision.md` task file.
